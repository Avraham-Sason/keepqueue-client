Keepqueue Client (Next.js 15) — TODO

Conventions
- Code style: CamelCase for variables/functions/components; Types/Interfaces in CamelCase
- Tech: Next.js 15 (App Router), TypeScript strict, Tailwind CSS, shadcn/ui, Origin UI, Zustand
- Data: Interact with Express API; Firebase Client SDK for Auth; Firestore direct reads minimized

1) Project Setup & Foundations
- Ensure Next.js 15 App Router structure: app/, components/, lib/, store/, hooks/, styles/
- Configure Tailwind CSS with shadcn/ui + Origin UI theme; global styles and dark mode
- ESLint + Prettier configuration aligned with project conventions
- Env loader for client (public vs server-side runtime keys)
- Set up absolute imports and path aliases

2) Auth & Session
- Integrate Firebase Auth (email/social/phone per PRD scope)
- Persist session and user claims; fetch custom claims (Owner/Staff)
- Route guards and server components that depend on auth
- Sign-in/Sign-up/Reset flows; Phone verification UI if enabled

3) Public Booking Flow (Customer)
- Business public page: branding, services list, ratings summary, reviews preview
- Service selection UI (durations, price)
- Date/Time picker that calls API GET /businesses/:business_id/availability
- Customer details form (name, phone/WhatsApp, notes, consent)
- Review & confirm screen; create appointment via API POST /businesses/:business_id/appointments
- Success page with details and WhatsApp confirmation note
- Customer portal: view upcoming/past appointments; reschedule/cancel (PATCH/POST cancel)
- Token-based cancel page path (for unauthenticated cancel link)

4) Smart Digital Calendar (Owner/Staff)
- Calendar views (day/week) showing staff assignments and services
- Create blocks/holidays and edit working hours
- Inline create/update appointment (owner/staff) with conflict feedback from API
- Availability visualization (computed via API)

5) Waitlist UX
- When no slots available: opt-in form to join waitlist
- Show customer’s waitlist status; allow removal
- Accept slot reservation link (TTL) from WhatsApp → claim via API

6) Business Admin Dashboard
- Profile & branding editor (logo upload to Firebase Storage, colors, public settings)
- Services CRUD (name, duration, price, active)
- Staff CRUD (name, role, working hours, breaks)
- Customers management (search, contact info, notes, flags)
- Shareable branded booking page link

7) Reviews & Ratings
- Post-appointment review form (rating, text)
- Display aggregated rating and reviews on business page
- Owner moderation tools (hide/report)

8) Reports & Analytics
- Dashboard with KPIs: bookings_by_period, no_show_rate, revenue_by_service, top_services
- Charts and filters by date range/service/staff

9) Problematic Customer Handling
- UI indicators for flagged customers; block booking with clear messaging
- Admin controls to override/unblock when needed

10) State Management & Data Layer
- Zustand slices: auth, business, services, staff, appointments, availability, waitlist, reviews, analytics
- API client module with typed methods; generate types from OpenAPI
- Request deduplication and optimistic updates where safe
- Error boundary components and toast notifications

11) Accessibility, Performance, SEO
- WCAG 2.1 AA: focus states, labels, roles, keyboard navigation
- Performance: code-splitting, image optimization, memoization, virtualization (lists)
- SEO: metadata, structured data on public business page, sitemap/robots

12) QA & DX
- Storybook for components (optional)
- E2E tests (Playwright/Cypress) for booking flow and admin basics
- Analytics events (page, CTA clicks, booking completed)

Deliverables
- Production-ready Next.js app that consumes the Express API
- Full booking flow, calendar, waitlist, reviews, analytics UIs implemented
- Strong UX with Tailwind + shadcn/ui and mobile-first responsiveness
